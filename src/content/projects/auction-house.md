## GUI에서 탐색하는 플레이어 거래소

플레이어는 복잡한 명령어 대신 거래소 GUI에서 현재 등록된 물품을 확인하고 구매할 수 있습니다. 각 매물에는 원본 아이템 정보와 함께 판매자, 전체·개당 가격, 등록 후 경과 시간, 구매 및 시세 확인 방법을 표시합니다.

한 화면에는 최대 45개의 매물을 표시하며, 검색어와 정렬 기준, 현재 페이지는 플레이어별로 분리해 관리합니다. 여러 플레이어가 동시에 서로 다른 조건으로 거래소를 탐색해도 화면 상태가 섞이지 않습니다.

### 검색과 가격 비교

최신·오래된 등록순, 전체 가격과 개당 가격의 오름·내림차순까지 여섯 가지 기준으로 정렬할 수 있습니다. 묶음 상품은 역직렬화한 아이템 수량으로 개당 가격을 계산해 실제 단가를 비교합니다.

검색 시에는 커스텀 표시 이름뿐 아니라 마인크래프트 한국어 아이템명, Material 이름, 네임스페이스 키를 함께 비교합니다. 공백, 밑줄, 하이픈과 영문 대소문자를 정규화해 입력 방식이 달라도 자연스럽게 매칭되도록 했습니다.

## 아이템 정보를 잃지 않는 판매 등록

Material과 수량만 저장하면 인챈트, 로어, 커스텀 모델과 플러그인 전용 데이터가 사라질 수 있습니다. 거래소는 실제 ItemStack 전체를 직렬화해 MySQL에 저장하므로 다음 정보를 판매 이후에도 보존합니다.

- 아이템 수량, 인챈트와 내구도
- 표시 이름, 설명과 커스텀 모델
- Persistent Data Container와 플러그인 전용 데이터

등록 가격은 숫자 형식과 최대 가격을 검증합니다. 데이터베이스 저장에 실패하면 먼저 제거했던 아이템과 등록 수수료를 다시 반환해 서버 오류로 플레이어의 재산이 사라지지 않게 했습니다. 전문 장비 시스템에서 소유자에게 귀속된 장비는 등록할 수 없도록 플러그인 간 데이터 규칙도 연결했습니다.

## 동시에 구매해도 한 명만 성공하는 구조

상품을 조회한 뒤 삭제하는 방식은 두 구매자가 거의 동시에 요청할 때 중복 구매나 아이템 복제로 이어질 수 있습니다. 이를 막기 위해 구매 시 **현재 상태가 ACTIVE인 매물만 SOLD로 변경**하는 조건부 갱신을 실행합니다.

데이터베이스에서 실제로 한 행을 변경한 요청만 이후 구매 절차를 계속합니다. 먼저 상태를 바꾼 구매자가 있다면 나머지 요청은 이미 판매되었거나 취소된 상품으로 처리됩니다. 매물 상태 변경과 거래 내역 생성은 하나의 데이터베이스 트랜잭션으로 묶어, 도중에 오류가 발생하면 매물이 잘못 판매 상태로 남지 않도록 롤백합니다.

### 결제 전 사전 검증

구매 버튼을 눌렀다고 바로 결제하지 않고 다음 조건을 먼저 확인합니다.

- 저장된 매물 데이터를 정상적으로 복원할 수 있는지
- 같은 아이템 스택의 남은 공간까지 포함해 인벤토리에 수용 가능한지
- 구매자에게 충분한 잔액이 있는지
- 매물이 아직 판매 가능한 상태인지
- 구매 처리 중에도 플레이어가 접속해 있는지

모든 조건을 통과한 뒤에만 경제 정산과 아이템 지급을 진행합니다.

## 중간 실패를 되돌리는 보상 거래

한 번의 구매에는 **매물 선점 → 판매자 대금 지급 → 판매 수수료 차감 → 완료 기록 → 구매자 아이템 지급**의 여러 단계가 필요합니다. 일부 단계만 성공하면 구매자와 판매자의 잔액 또는 아이템 상태가 서로 맞지 않게 됩니다.

거래소는 단계별 보상 처리를 구현했습니다. 판매자 대금 지급 후 수수료 차감에 실패하면 지급액을 회수해 구매자에게 돌려주고, 경제 정산 후 완료 기록에 실패하면 대금과 수수료를 보상한 뒤 매물을 다시 활성화합니다.

정산 금액은 `BigDecimal`로 변환해 소수점 오차를 줄였습니다. 각 지급·수수료·환불에는 매물 ID를 포함한 고유 사유를 남겨 거래 단위로 추적할 수 있습니다. 오류를 알리는 데서 끝나지 않고, 이미 진행한 작업을 반대 거래로 되돌려 데이터와 재화를 원래 상태에 가깝게 복구하는 구조입니다.

## 오프라인 판매자 정산과 거래 기록

판매자가 접속해 있지 않아도 구매와 판매 대금 지급은 정상적으로 진행됩니다. 완료된 거래에는 원본 매물 ID, 판매자·구매자 UUID, 아이템 데이터, 판매 가격과 수수료, 실수령액, 완료 시간, 정산·알림 상태를 기록합니다.

판매자가 다시 접속하면 확인하지 않은 거래를 조회해 판매 가격, 수수료와 실수령액을 안내합니다. 알림 조회도 트랜잭션과 상태 값으로 관리하며, 처리 도중 플레이어가 접속을 종료하면 확인 상태를 되돌려 다음 접속에서 다시 받을 수 있게 했습니다.

## 실제 거래를 이용한 시세 분석

플레이어가 손에 든 아이템의 적정 가격을 판단할 수 있도록 최근 7일의 실제 완료 거래와 현재 매물을 함께 분석합니다.

- 최근 거래의 평균가, 중앙값과 추천 시세
- 거래 표본 수와 표본 수에 따른 신뢰도
- 현재 등록 매물 수와 최저 전체·개당 가격
- 해당 아이템의 최근 거래 내역

추천 시세는 극단적으로 높거나 낮은 가격의 영향을 덜 받는 중앙값을 우선합니다. 표본이 없을 때는 임의의 가격을 만들지 않고 거래 데이터가 부족하다는 사실을 명확히 표시합니다.

### 전문 장비의 능력치까지 반영

같은 이름의 전문 장비라도 직업과 전문력이 다르면 가치가 달라질 수 있습니다. PDC에 저장된 직업 종류와 전문력을 읽어 같은 직업이면서 현재 장비와 전문력이 가까운 거래 기록을 먼저 보여줍니다. 예를 들어 채광력 700인 곡괭이는 이름만 같은 모든 장비가 아니라 능력치가 비슷한 곡괭이의 거래를 우선 참고합니다.

## 만료 상품의 재시도 가능한 반환

기본 설정에서는 등록 또는 마지막 갱신 후 14일이 지나면 매물이 만료됩니다. 만료가 가까운 상품은 판매자 접속 시 미리 알리고, 판매자는 자신의 판매 목록에서 무료로 기간을 갱신할 수 있습니다. 가격 변경 시에도 판매 기간과 경고 상태를 함께 초기화합니다.

만료된 아이템은 즉시 지급하거나 삭제하지 않고 서버의 영구 우편함 시스템으로 반환합니다. 먼저 `ACTIVE → EXPIRING` 상태로 처리 대상을 선점하고, 우편 전달에 성공한 경우에만 `MAILED`로 확정합니다. 변환이나 전달에 실패하면 완료 상태로 바꾸지 않아 다음 정리 작업에서 다시 시도할 수 있습니다.

우편에는 매물 ID 기반의 고유 전달 키를 사용해 재시도 과정에서 같은 아이템이 중복 지급될 가능성도 줄였습니다.

## 판매 관리와 경제 설정

판매자는 별도 GUI에서 자신의 전체 매물을 확인하고 기간 갱신, 가격 변경, 판매 취소와 아이템 회수를 진행할 수 있습니다. 다른 플레이어의 이름을 입력해 해당 판매자의 상품만 둘러보는 기능도 제공합니다.

등록·판매 수수료율과 최소·최대 등록 수수료, 고액 거래 기준 및 별도 수수료율, 최대 등록 가격은 설정 파일에서 조정할 수 있습니다. 등록 수수료가 없는 유저 상점부터 등록 단계에도 비용이 드는 경매장까지 서버 경제 정책에 맞게 구성할 수 있습니다.

## 게임 틱과 데이터베이스 작업의 분리

MySQL 연결은 HikariCP 커넥션 풀로 관리합니다. 상품 등록과 페이지 조회, 검색, 거래 기록, 시세 계산, 만료 정리 같은 데이터베이스 작업은 비동기 스레드에서 실행하고, GUI를 열거나 플레이어에게 결과를 전달하는 Bukkit API 작업은 다시 메인 스레드로 전환합니다.

Citizens가 설치된 환경에서는 지정한 NPC를 거래소와 연결해 우클릭으로 GUI를 열 수도 있습니다.

## 검증

정상 거래뿐 아니라 경제 정산의 중간 실패와 만료 설정을 자동 테스트로 검증했습니다.

- 판매 대금과 수수료의 정상 정산
- 판매 수수료 차감 실패 시 지급 대금 회수
- 보상 거래 실패 상태의 호출자 전달
- 매물 만료 시각 계산
- 잘못된 만료·경고 설정값 보정

현재 준비된 **자동 테스트 5개가 모두 통과**합니다.
